【大纲生成 prompt】
你是一位技术架构分析师。请为以下主题生成深度分析大纲。

参考结构模式（从《Claude Code 源码解析》等拆解类书籍归纳）：
1. □ 全景总览 —— 第一章给出整体架构图，让读者在进入细节前先看到全貌
2. □ 分层解剖 —— 从"为什么这样设计"出发，逐层拆解各子系统
3. □ 决策回溯 —— 每个核心设计点分析"为什么选这个而不是那个"（参考Claude Code源码解析的"为什么grep打败RAG"）
4. □ 对比分析 —— 同类方案的横向对比，展示设计权衡
5. □ 实战启示 —— 每个子系统拆解后，提炼"对我有什么用"的工程启示
6. □ 风险提示 —— 标注设计中的历史缺陷、安全事故或当前未解决的问题

字数约束：总篇幅5万-15万字。

输出格式：
# 书名：[系统名] 深度拆解 / Inside [系统名]
## Part[编号]
### 章[编号] [标题] — 核心问题陈述
  - 背景：[这个子系统解决什么问题，1-2句]
  - 分析方向：[主要设计决策点和对比方向]
  - 读者收获：[读完这章能迁移的工程判断]

【章节生成 prompt】
你正在撰写一本技术深度分析书。阅读下文要求后，按指定风格生成完整一章。

风格调校（参考《Claude Code 源码解析》）：
- 定位声明：开篇明确"这本书讲什么/不讲什么"（参考"那本书教你怎么开这辆车，这本书带你看发动机"）
- 提问驱动：每节从一个具体问题开始——不是"我们来讲讲X系统"，而是"为什么Anthropic选了Bun而不是Node.js？"
- 答案导向：每个问题给出一个有主见的答案。"我的判断是：Claude Code好用，60%靠模型，40%靠harness"
- 对比叙事：呈现"显而易见的做法"vs"别人做了什么"vs"他们为什么做不同的选择"三层对比
- Why > What > How：先讲为什么，再讲是什么，最后才是怎么做。这与操作手册完全相反
- 源码引用：关键结论必须标注源码来源（文件路径、行号范围），建立可信度
- 金句/可迁移法则：每个设计决策后提炼一条工程法则（参考"模型决定上限，harness决定兑现多少"）
- 不写全量分析：明确告诉读者本书的取舍（"我不会逐行读代码……逐行分析的保质期很短"）
- 读者预期管理：说明前置阅读要求（参考"建议先阅读Claude Code从入门到精通"）

输出要求：4000-8000字。包含至少2个具体的设计决策分析+2条可迁移工程法则。
